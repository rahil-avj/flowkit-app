## Findings

### Critical

**F1 — `npx tsc --noEmit` fails on a freshly scaffolded project, before any custom code is written, and continues to fail on every file the CLI itself generates.**
- **Where:** every screen file generated by `create:screen` (new: `flows/checkout/cart/CartScreen.tsx`, `flows/checkout/confirmation/ConfirmationScreen.tsx`, `flows/nonexistent/foo/FooScreen.tsx`; pre-existing scaffold: `flows/home/detail/DetailScreen.tsx`, `flows/home/home/HomeScreen.tsx`, `flows/onboarding/{ready,setup,welcome}/*.tsx`); also `vite.config.ts` itself.
- **Root cause:** newly generated screens import `type { FlowScreenProps } from '@platform/types'`; pre-existing scaffolded screens import `{ FlowScreenProps }` from bare `'flowkit'`. Neither resolves under plain `tsc`:
  - `@platform` is not declared as a path alias in the scaffolded project's own `tsconfig.json` — it is only injected as a **bundler-time** alias by the `flowkit/vite` Vite plugin (`node_modules/flowkit/scripts/helpers/vite-plugin.js`, `config()` hook, `'@platform': engineSrc`). `tsc` has no visibility into Vite plugin-injected aliases.
  - `flowkit`'s package `exports["."].types` points only to `./dist/types/core/config/index.d.ts`, which does not re-export `FlowScreenProps` — that type lives in `dist/types/types/index.d.ts`, a subpath not exposed in the public `exports` map at all.
  - `vite.config.ts`'s own `import { flowkit } from 'flowkit/vite'` has no `.d.ts` for that subpath export, and the project's `tsconfig.json` has no `declare module 'flowkit/vite'` shim, so it's implicitly `any` under `strict: true`.
- **Verified independently:** `npm run build` (real `vite build`) succeeds completely cleanly on the exact same file set — 2007/2008 modules transformed, zero errors, only the known-acceptable >500KB chunk warning. This proves the aliases and exports genuinely do resolve at real build time; `tsc --noEmit` is giving false-positive-looking errors relative to the actual, working build.
- **Impact:** any agent or human trusting a bare `npx tsc --noEmit` as a correctness gate on this project — which is exactly what this study's own task list, and by extension any reasonable CI/pre-commit setup, would naturally reach for — will conclude the project is broken when it demonstrably is not. This is the single most consequential finding of the run: a scaffolded flat-mode FlowKit project does not typecheck standalone out of the box.
- **Post-hoc correction (independently re-verified by the orchestrating session after this report was drafted):** a plain `npx tsc --noEmit` from the project root, in the current post-conversion (multi-workspace) state, reports only the `vite.config.ts`/`flowkit/vite` declaration error (TS7016) — it does not print the `@platform/types` errors in that same invocation, because tsc stops at that error before reaching the screen files. The `@platform/types` resolution failure is still real and independently reproducible (confirmed via an isolated tsconfig scoped to a single screen file: `Cannot find module '@platform/types'`, TS2307) — but the two errors are not simultaneously visible in one default run against this project's real `tsconfig.json`. Treat F1 as two related-but-distinct sub-findings, not one unified error list: (a) `vite.config.ts` has no type declarations for `flowkit/vite`'s exports, and (b) `@platform`/`@workspace` aliases are invisible to `tsc` project-wide. Fixing (a) alone would very likely surface (b) in the same `tsc --noEmit` run for the first time — so both need addressing, and (a) should probably be fixed first simply to unmask (b).
- **New/recurring/regression:** new (no prior report exists to compare against).
- **Severity:** Critical — actively misleads both agents and humans about project health using the single most standard verification command in the entire JS/TS ecosystem.

### High

**F2 — `create:component`'s own generated usage hint (both printed to stdout and baked into the file as a comment) is a named import for a default export.**
- **Where:** `flowkit create:component --name:PriceTag --path:lib/components/ui` output: `Usage: import { PriceTag } from '@workspace/lib/components'`. Same string is written verbatim as a comment into the generated `lib/components/ui/PriceTag.tsx` (line 2).
- **Ground truth:** the generated file uses `export default function PriceTag(...)` — a default export, not a named one. As literally printed, `import { PriceTag } from '@workspace/lib/components'` would fail on two independent counts: (1) `PriceTag` is not a named export, and (2) `@workspace/lib/components` isn't even a real path — no barrel `index.ts` exists at `lib/components/` (the CLI's own preceding line acknowledges this: `No index.ts found — add export manually or run: flowkit add:export`). The only import that would actually work is `import PriceTag from '@workspace/lib/components/ui/PriceTag'` (default import, full file path).
- **Impact:** an agent following the CLI's own printed guidance literally writes code that fails to compile. Because the wrong guidance is also baked into the source file itself as a persistent comment, it keeps misleading every future reader, not just the one session.
- **New/recurring/regression:** new.
- **Severity:** High — directly, repeatedly wrong guidance from the tool's own output, compounding because it's persisted into checked-in source.

**F3 — `@workspace` alias, like `@platform`, is invisible to `tsc` for the same structural reason as F1.**
- **Where:** same mechanism as F1, applied to the component/barrel-import guidance in F2.
- Folded into F1's root cause but called out separately because it affects a different command family (`create:component` vs. `create:screen`) — same fix would need to cover both.
- **Severity:** High (subsumed under F1's fix, not double-counted in overall severity).

### Medium

**F4 — `create:component`'s required `--path` flag is not mentioned in `flowkit help`'s summary line.**
- **Where:** `flowkit help` output: `create:component --name:<id>        Add a workspace component` — no `--path` shown.
- **Actual behavior:** running `flowkit create:component --name:PriceTag` alone fails with exit code 1: `✗ --name:<PascalName> and --path:<lib/components/...> are required`, immediately followed by a correct, complete example. Recovery was trivial (one retry, correct on the second attempt) precisely because the *error message* (not the help summary) carried the real usage.
- **Impact:** low-cost but real gap between the top-level command reference and actual required arguments; the only reason this didn't cost more time is that the error text itself was excellent.
- **New/recurring/regression:** new.
- **Severity:** Medium — friction, not breakage; fully mitigated by good error-message design (see F-working-well below).

### Low / Working-well (see full section below)

**F5 — `add:step`'s flag name (`--action`) doesn't match the field it populates (`actionNote`).**
- **Where:** `flowkit add:step --flowplan:checkout --screen:cart --action:"..."` → written into the flowplan file as `actionNote: "..."`.
- **Impact:** trivial, discoverable immediately from the diff, never blocked anything. Noted only because docs/CLAUDE.md's flowplan step shape is documented as `{ screenId, on, actionNote }`, and the CLI flag doesn't echo that name 1:1. Cosmetic per user's own stated review priorities, but flagged since it's an objective naming mismatch, not an opinion.
- **Severity:** Low.

- **Does a real, isolated `general-purpose` sub-agent (per the skill's intended Step 2) surface additional or different confusion than this in-process run?** This run's procedural deviation (no sub-agent spawn tool available) means the acting agent retained awareness of being "the study," which could subtly reduce genuine first-contact confusion compared to a truly fresh, isolated agent. - **Is F1 (tsc false positives) specific to `create-flowkit-app`'s flat-mode scaffold, or does it also affect `create-flowkit-workspace`'s multi-workspace scaffold from a cold start (not via `convert:multi`)?** This run only exercised flat→multi conversion, not a project scaffolded directly into multi-workspace mode. A future run should scaffold via `create-flowkit-workspace` directly and re-check tsc/build parity.
- **Would adding `@platform`/`@workspace`/etc. as real `tsconfig.json` `"paths"` entries in the scaffolder templates (pointing at whatever `tsc`-visible location, e.g. re-exporting through the published package's actual type surface) close the F1 gap without touching the Vite plugin at all?** Plausible surgical fix, not evaluated here — this skill's job ends at reporting, not fixing.
- **No ESLint config ships with `create-flowkit-app` at all** — worth a deliberate product decision (ship a default flat config, or explicitly document "bring your own lint config") rather than leaving it as a silent gap discovered only when someone tries to lint.
