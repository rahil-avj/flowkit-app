# Changelog — Interactive Screens Preview setting

Date: 2026-07-12

## Context

User report (multi-workspace consumer project, `@rahil316/flowkit@0.0.0-canary.1`): buttons on a screen viewed via the left panel's **Screens tab** do nothing — no navigation, no error. Traced to `PreviewCanvas.tsx`'s Screens-tab render path, which only ever passed `db` to the screen component; `onAction`/`onNext`/`onBack` were `undefined`, so any `onClick={() => onAction?.(...)}` silently no-op'd via optional chaining.

Investigation confirmed this was intentional (documented in `FlowScreenProps`'s JSDoc) and not a regression (only one commit in history has ever touched that render path). The user asked for it to be interactive instead, with a settings toggle rather than a default behavior change.

## What changed

Added a new, **default-off** setting: **Settings → Interface → "Interactive Screens preview"**.

- Off (default): no behavior change from today — Screens tab remains a static, click-inert preview.
- On: screens viewed from the Screens tab get a working `onAction` callback wired to real navigation (`navigateTo`), so buttons using the direct-id navigation convention (`onClick={() => onAction?.('some-view-id')}`) work without entering Flows/playback mode.

**Known limitation, by design:** `onNext`/`onBack` remain inert even with the setting on. Outside flowplan playback there is no sequential screen order or interaction-rule map to resolve "next"/"back" against — a screen can be listed standalone or referenced by multiple flows, so there's no single canonical position to advance from. Only `onAction(name)` is addressable, and only when `name` resolves to a real view id.

## Files touched

| File | Change |
|---|---|
| `src/shared/constants/storageKeys.ts` | Added `LS_INTERACTIVE_SCREENS_PREVIEW` key |
| `src/core/canvas/PreviewCanvas.tsx` | New `interactiveScreensPreview` state (localStorage-backed via `useState` initializer, same pattern as `autoHideScrollbars`); plumbed through `CanvasContent`; when on, `ActiveComponent` receives `{ db, isFlow: false, onAction: navigateTo, onNext: noop, onBack: noop }` instead of just `{ db }` |
| `src/shared/components/mobile/MobileCanvas.tsx` | Mirrored the same toggle + render wiring for the mobile canvas surface, so behavior/settings stay consistent between desktop and mobile |
| `src/shared/components/overlays/appActions.ts` | Added `interactiveScreensPreview` / `toggleInteractiveScreensPreview` to `ActionCtx` |
| `src/shared/components/overlays/Settings.tsx` | New toggle row under Settings → Interface, with hint text explaining the `onNext`/`onBack` limitation |

- `npx eslint` on all touched files — clean

- Not manually verified in a running browser session — no workspace was scaffolded in this checkout (`src/workspaces.json` had `active: null`, no `workspaces/` content) to click through live. Recommend a manual pass in a real workspace before merging/publishing.

## Not in scope

- Making `onNext`/`onBack` functional from the Screens tab — would require resolving "the next screen" from `screenOrder`/flow membership for a screen that may belong to zero, one, or multiple flows. Flagged to the user as a larger follow-up if needed.
- Mobile's pre-existing gap where `ActiveComponent` never received `db` at all (flow or no flow) — separate, unrelated issue, not touched here.
