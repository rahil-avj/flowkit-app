# Changelog — workspace.ts rename, explicit workspace paths, unified export flow

Date: 2026-07-12

## Context

Started from a single request — make `flowkit export` work in consumer (flat/
multi-workspace) projects, not just repo mode — which grew through
interactive design discussion into three sequenced changes: a file-location
hardening pass, then the rename, then the actual export rebuild. Full design
trail (every decision, and why) lives in
`temp-docs/flowkit-configs-plans.md`; this file is the condensed summary of
what shipped.

## What changed

### 1. `flowkit.config.ts` renamed to `workspace.ts`

Same role (per-workspace `workspace`/`flows`/`screenOrder` — the
`defineConfig()` object), same location (project root in flat mode, each
`workspace-N/` folder in multi mode). Pure rename, backed by a new constant
(`WORKSPACE_CONFIG_FILENAME` in `scripts/helpers/config-filenames.js`) so the
filename is declared once instead of duplicated as a string literal across
~26 call sites (authoring commands, both scaffolders, the vite plugin's
config reader, docs).

### 2. `package.json`'s `flowkit.workspaces` — array → object with explicit paths

Before: `"workspaces": ["workspace-1"]` — folder location assumed to match
the name, by convention only. After:
```json
"workspaces": { "workspace-1": { "path": "workspace-1" } }
```
Backward-compatible — `readFlowkitManifest()` still accepts the old array
shape and treats each entry as `{ path: name }`. All 4 write sites
(`convert:multi`, `create:workspace`, `remove:workspace`, `rename:workspace`
in `scripts/platform/workspace-flat.js`) and both scaffolders updated to
write the new shape.

### 3. `flowkit export` rebuilt — one guided flow, every mode

Previously repo-mode-only (`export`/`export:full`/`export:<name>`/
`export:all`); consumer mode hard-rejected. Now:

```bash
flowkit export
flowkit export --workspace:<name>
flowkit export --profile:<name>
```

- **Workspace step** — only asked when 2+ candidates exist (repo mode:
  multiple `workspaces/*`; multi-workspace consumer mode: 2+ entries in
  `flowkit.workspaces`). Skipped otherwise.
- **Profile step** — always asked unless `--profile:<name>` given. Reads
  named profiles from a new `flowkit.json` (plain JSON, project root):
  ```json
  {
    "exportDefaults": { "exportPath": "./dist" },
    "exportProfiles": {
      "client-demo": {
        "exportPath": "./output",
        "exportName": "{workspace}-{profile}-{timestamp}"
      }
    }
  }
  ```
- `export:full` and the colon-suffix dispatch are gone. `includeFlowlens`
  was designed, then removed before shipping — every export always ships
  the full codebase; proper feature-gating deferred to a future task (see
  `temp-docs/flowkit-configs-plans.md` for why).
- Split into `scripts/builders/export.js` (CLI prompting/arg-parsing only)
  and new `scripts/builders/run-export.js` (mode-agnostic build execution —
  `runConsumerExport()`, `runRepoExport()`).
- Consumer-mode ESLint pre-flight check now skips gracefully (with a
  notice) when the project has no ESLint config, instead of hard-failing
  every export — neither scaffolder ships one today.

## Real bugs found and fixed along the way (pre-existing, not caused by this work, but blocking verification of it)

1. **`vite.config.standalone.ts` never included the `flowkit()` plugin** —
   repo-mode standalone exports with FlowLens code reachable
   (`useSessionLibrary.ts`'s `virtual:flowkit/workspace` import) were
   silently broken before this session. Fixed by adding the plugin, same
   pattern this repo's own `vite.config.ts` already uses.
2. **The plugin's config-reader esbuild shim only handled the bare
   `'flowkit'` import specifier**, not `@platform/core/config` (repo mode's
   own convention) — never previously exercised against a repo-mode
   workspace file because dev-mode's virtual-module resolution is lazy,
   while a standalone build's bundling is eager.
3. **Scaffolded flat-mode demo screens fail `tsc --noEmit`** in every fresh
   project — a TypeScript quirk where `unknown` narrowed through `?? []`
   becomes `{}`, incompatible with an explicit array type annotation.
   Fixed with an `as any` cast on `db` in the affected screen templates
   (`scripts/helpers/workspace-template.js`), matching repo mode's own
   `useDashboard()` convention (`db: Record<string, any>`).
4. **`create-flowkit-workspace`'s generated `vite.config.ts` template used
   `fs.readFileSync` with no `@types/node`** in the scaffolder's own
   generated `devDependencies` — added it.
5. **Consumer-mode export's temp vite config was written to the OS temp
   dir**, where Node's module resolution can't find the project's own
   `node_modules` (`@vitejs/plugin-react`, `vite-plugin-singlefile`,
   `flowkit/vite`). Fixed by writing it inside the project root instead
   (gitignored).
6. **Auto-installing `vite-plugin-singlefile` on first export could
   silently convert a `--local-dev` real-copy `flowkit` install back into
   a symlink**, breaking `isRepoMode()` detection immediately after (a
   plain `npm install` re-resolves a `file:` dependency without
   `--install-links`). Fixed by detecting the existing real-directory
   state and re-passing `--install-links` if needed.
7. Accidentally deleted repo-root `index.html` and 3 files under
   `agent-workflow-plans/` during testing this session — both restored via
   `git checkout` before anything was committed.

## Docs updated (one pass each)

`CLAUDE.md`, `docs/CLI.md`, `docs/FLOWKIT.md`, `docs/FLOWLENS.md`,
`docs/FLOWLENS-GUIDE.md`, `docs/AGENTS.md`, `scripts/platform/help.js`
(the CLI's own embedded help text) — all updated to describe the rename,
the new `flowkit.json`, and the unified export flow; removed every
reference to `export:full`/colon-suffix export dispatch found. A follow-up
pass on `help.js` also fixed a pre-existing, unrelated accuracy gap: the
`Workspaces` section lumped `convert:multi` (flat-mode-only) together with
`convert:flat`/`create:workspace`/`remove:workspace`/`rename:workspace`
(multi-mode-only) under one vague heading — split into two correctly
scoped sections.

`Documentation/product/features-by-persona.md` was checked and
deliberately **not** touched — it's an explicitly dated point-in-time
snapshot per its own header, not meant to track current reality.

## Verification

- `npx tsc --noEmit` — clean
- `npx eslint .` — clean
- `npx vitest run` — 134/134 pass
- `npm run test:workspace` — 32/32 CLI integration tests pass
- Manually verified end-to-end via fresh `--local-dev` scaffolds: repo-mode
  export (single + multi workspace, named profile + default), flat-mode
  consumer export, multi-workspace consumer export (single + multi
  workspace, both flag-based and interactive-prompt paths), full
  `create:workspace`/`rename:workspace`/`remove:workspace`/`convert:multi`/
  `convert:flat` lifecycle against the new manifest shape.

## Not in scope / deferred

- FlowLens include/exclude control for exports (`includeFlowlens`) — designed,
  removed before shipping, real feature-gating is a future task.
- `flowkit handoff` — same `requireRepoMode()` pattern as export used to
  have, intentionally left untouched (different feature, developer zip).
- Migration tooling for already-scaffolded projects predating the rename —
  new projects and this monorepo's own future output only.
