# FlowMaster — Scope

FlowMaster is the flow execution engine. Its job is to take an authored flow definition and execute it deterministically — handling screen sequencing, interaction routing, guards, animations, state, and the analytics event stream. Screens are dumb; FlowMaster is smart.

---

## In scope

### Execution engine

- `useFlowEngine()` — interaction state machine; manages `activeScreenIndex`, history, `animClass`, transition commits, guard evaluation
- `FlowMaster.tsx` — renders a single flow; consumes `useFlowEngine()`; applies flowplan gating, swipe gestures, hotspots, off-script toasts
- `ANIM_DURATION = 280ms` — screen transition timing; injected dynamically, not a Tailwind class

### Authoring formats (both compile to the same runtime)

- **Old** (`_playFlow.ts`): `FlowDef` with `id`, `label`, `screens[]`, `on:` map, `interactions:`, `localData`, `canEnter/canNotEnter/canEnterFallback`, `meta`
- **Flowplan** (`defineFlow`): `FlowplanDef` with `id`, `name`, `steps[]` (each with `screenId`, `on?`, `db?`, `actionNote?`, `forks?`), `simulator?`

### Compilation

- `compileFlowplan.ts` — pure/testable; converts `FlowplanDef` → `FlowConfig` (runtime) + `CompiledStep[]` (playback gating + db patches)
- `normalizeOnMap()` — converts `on:` shorthand to verbose `interactions:` shape at runtime
- Runtime never depends on the authoring format — `DevelopmentValues.md` Principle 2

### Interaction types

- Triggers: `tap`, `double-tap`, `long-press`, `hover`, `swipe-left`, `swipe-right`, `swipe-up`, `swipe-down`
- `goTo` targets: screen id, `"next"`, `"back"`, `"__complete__"`, or a resolver function `(ctx) => string`
- `do`: `(ctx: InteractionCtx) => void` — db mutation before navigation
- `with`: animation class; `delay`: ms before navigation fires

### `InteractionCtx` (full shape)

`activeScreenId`, `history`, `flowState`, `get(key)`, `set(key, value)`, `db`, `updateDb(updater)`, `effect(name)`

### Guards

- Flow-level: `canEnter`, `canNotEnter`, `canEnterFallback`
- Screen-level: `canEnter`, `canNotEnter` in `screenMeta`
- Both evaluated; either blocking = guard fails

### Screen features

- `autoAdvanceDelay` — auto-advance to next screen after N ms
- `enterAnimation` — animation played when navigating TO this screen
- `hotspots` — clickable `{id, x, y, w, h}` regions in % of screen; wired via interactions map
- `isStandalone` in `screenMeta` — marks a screen as an entry point (not reached via back-nav)

### Flow-local state

- `localData` in `_playFlow.ts` — per-flow sandbox; reset on flow exit
- Accessible as `flowState` in screens and `ctx.get()`/`ctx.set()` in interaction resolvers

### Analytics event stream

- FlowMaster emits events to FlowTracer on every navigation: `flow.entered`, `flow.transition`, `screen.visited`, `screen.dwell-end`, `interaction.*`, `navigation.auto-advance`
- `flow.transition` carries `{ action, from, to, blocked?, error?, warnings[] }` — records why a tap misbehaved
- Emitting is fire-and-forget; FlowTracer is an observer — it never changes execution

### Flowplan playback (F4/F5)

- `FlowPlaybackContext` — gates interactions to compiled steps; applies db patches at each step
- `enter()` / `exit()` / `restart()` / `applyStep()` — playback lifecycle
- Seam left for Phase 2 visual editor: "Phase 2 (visual editor) keep extending"

## Out of scope

- Canvas rendering, zoom, pan — that is FlowKit core
- Session recording infrastructure — that is FlowLens/FlowTracer
- Analytics aggregation and replay — that is FlowLens
- The flow debugger UI panel — that is the `flow-debugger` feature (consumes FlowMaster's event stream)
- `router.tsx` in old workspaces — auto-generated by the CLI, not by FlowMaster

---

## Key constraints

- **The runtime never reads authoring formats.** Compilation is a one-way boundary (Principle 3).
- **Screens never import routing logic.** Navigation is wired in `_playFlow.ts` or the flowplan — never in screen component code.
- **`useFlowNav()` not `navigateTo` from `useDashboard()`.** Direct `useDashboard().navigateTo` in a screen bypasses FlowMaster guards, animations, and the recorded `flow.transition` event.
- **`ANIM_DURATION = 280ms` is not replicated in UI components.** It is injected dynamically by FlowMaster only.
